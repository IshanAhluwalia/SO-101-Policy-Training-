# Session Log — 2026-06-15

## What Was Done
- Got teleop running for the first time in this setup
- Diagnosed and fixed multiple issues blocking teleop

## Issues Encountered & Fixed

### Leader arm not connecting
- Port `/dev/tty.usbmodem5B3D0458801` existed but no motors found
- Fixed by replugging USB cable on leader arm

### Motor 5 (wrist_roll) overload on follower
- `[RxPacketError] Overload error!` caused full bus failure
- Fixed by full power cycle (barrel jack, not just USB) and moving arm to relaxed position

### sync_read failures (`no status packet` / `incorrect status packet`)
- Teleop crashed after 1 frame on `sync_read("Present_Position")`
- Root cause: USB bus contention at 60Hz with 6 motors
- Fixed by adding `num_retry=10` to sync_read calls in:
  - `src/lerobot/robots/so_follower/so_follower.py` (lines 181, 215)
  - `src/lerobot/teleoperators/so_leader/so_leader.py` (line 148)

### Follower continuously moving without leader input
- Root cause: no calibration — raw encoder values differ between arms
- Workaround: added `--robot.max_relative_target=5` to teleop.sh

### wrist_flex snapping to 2 extreme positions
- Root cause: calibrated range was 0–4095 (physical limits not detected)
- Partial fix: ran calibration on both arms but wrist_flex still problematic

### wrist_flex leader encoder randomly resets to 0
- Happens mid-range, not at physical limits
- Likely loose JST encoder cable inside motor 4 on leader arm

## Decisions Made
- `max_relative_target=5` added to teleop.sh as permanent workaround until proper calibration
- Patch file `lerobot_fixes.patch` exported for reproducibility

## State at End of Session
- Teleop runs but wrist_flex calibration is still bad
- Both arms need recalibration with care around hard stops
